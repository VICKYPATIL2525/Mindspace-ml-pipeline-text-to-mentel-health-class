  With 7 classes and moderate imbalance, f1_weighted is dominated by the

*** CRITICAL ISSUE #7 — DATA LEAKAGE: Outlier handling before train/test split ***
  The PowerTransformer (yeo-johnson) is FIT on the ENTIRE dataset (50,000 rows)
  including what will become the test set. The IQR bounds for winsorizing are
  also computed on the full dataset. This means:
    - Test data statistics influence the transformation parameters
    - The model indirectly "sees" test data patterns during training
    - Reported test metrics will be OPTIMISTICALLY BIASED
  This is the #1 most damaging issue in the entire pipeline.
  FIX: Move outlier handling AFTER the train/test split (Step 12), or
  fit transformations only on training data and apply to both train and test.

ISSUE #8 — Yeo-Johnson fitted objects not saved
  The PowerTransformer objects are created inside the loop and discarded.
  At inference time on new data, you cannot reproduce the same transformation
  because you don't have the fitted lambda parameters.
  FIX: Store each fitted PowerTransformer in a dict and save with artifacts.

ISSUE #9 — Strategy always minimizes skewness, but skewness != best for models
  Tree-based models (RF, XGBoost, LightGBM) do NOT care about skewness —
  they handle any distribution natively. Aggressively normalizing distributions
  with yeo-johnson can actually DESTROY useful signal patterns.
  The output shows yeo-johnson won for almost every column, meaning most
  features were transformed with a complex power transform unnecessarily.
  SUGGESTION: Only apply outlier smoothing for linear models, or make it
  optional. For tree models, raw features or simple winsorizing is better.

ISSUE #11 — drop_first=True hurts tree-based models
  One-hot encoding with drop_first=True drops the reference category

  This is needed for linear models to avoid multicollinearity, but for
  tree-based models (which dominate this pipeline), it LOSES information.
  A tree model cannot easily represent "if language == english" with just
  "not hindi AND not marathi".
  SUGGESTION: Use drop_first=False for tree-based models.

*** ISSUE #14 — DATA LEAKAGE: Feature selection on full dataset ***
  Random Forest importance and Mutual Information are computed on ALL 50,000

*** ISSUE #15 — VIF removed important features! ***
  VIF removed 3 features, including:
    - future_focus_ratio (VIF=48.4) — was ranked #3 by Kruskal-Wallis (H=20530)
    - overall_sentiment_score (VIF=15.8) — was ranked #2 by Kruskal-Wallis (H=24571)
  These were among the TOP predictive features! VIF measures multicollinearity
  (redundancy between FEATURES), not importance. Removing a highly predictive
  feature just because it's correlated with other features is a BAD trade-off.

  What happened: overall_sentiment_score is likely a combination of
  positive_emotion_ratio and negative_emotion_ratio (hence high VIF), but it
  captures target-relevant signal that neither component captures alone.

  SUGGESTION: Either skip VIF entirely (tree models handle multicollinearity
  natively) or only remove VIF-flagged features if they rank BELOW median
  in the importance ranking. Never remove a top-10 important feature for VIF.

ISSUE #16 — Bottom 30% removal is too aggressive

  SUGGESTION: Use a threshold on absolute score (e.g., MI > 0.01) instead
  of a fixed percentage.

ISSUE #21 — Tuning uses 3-fold CV but training used 5-fold CV
  The tuning CV (3-fold) produces different score estimates than the training
  CV (5-fold). This means the tuned "best_score" is not directly comparable
  to the training "mean_score". A model might appear to improve from tuning
  when it's actually just a different CV split producing a different estimate.
  SUGGESTION: Use the same number of folds (5) for both, or at least note
  this discrepancy in the output.

    ✗ Which outlier strategy was applied to which column
    ✗ Frequency encoding maps (if any were used)
    ✗ One-hot encoding categories (to handle unseen categories)
    ✗ Feature list AFTER selection (what columns to expect)
    ✗ Column order mapping

  Without these, you CANNOT process new data through the same pipeline.
  The saved model expects 42 features in a specific order with specific
  transformations, but only the scaler is saved.


================================================================================
SUMMARY: WHAT THE PIPELINE DOES WELL
================================================================================

1. TARGET LEAKAGE DETECTION (Step 5)
   Caught the "target" column as leakage — this is excellent and would have
   completely ruined the model without detection.

2. CONDITIONAL MODEL SELECTION (Step 13)
   Smart dynamic selection based on dataset size/dimensionality. Correctly
   skipped SVM (too slow for 40K samples).

3. MULTI-METHOD CONSENSUS FEATURE SELECTION (Step 11)
   Using 4 independent methods (RF, MI, Kruskal-Wallis, Levene's) and averaging
   ranks is a robust approach that doesn't over-rely on any single method.

4. ADAPTIVE OUTLIER HANDLING STRATEGY (Step 8)
   Testing multiple strategies per column and picking the best one is smart.
   (Though the execution has leakage issues, the IDEA is good.)

5. CLASS IMBALANCE HANDLING (Step 7)
   Correctly identifies moderate imbalance and decides on class_weight strategy.

6. COMPREHENSIVE EDA (Step 10)
   Multiple statistical tests, correlation analysis, distribution analysis,
   and visualizations provide good data understanding.

7. OPTUNA TUNING WITH TIMEOUT (Step 16)
   Bayesian optimization with a hard timeout prevents the tuning from running
   forever. Good pragmatic choice.


================================================================================
SUMMARY: WHAT THE PIPELINE DOES WRONG
================================================================================

*** CRITICAL ISSUES (will produce wrong/inflated results) ***

1. DATA LEAKAGE — Train/test split too late (Steps 8, 9, 10, 11)
   Outlier handling, encoding, EDA, and feature selection all happen BEFORE
   the 80/20 split. Every transformation is fitted on the full dataset.
   IMPACT: Test metrics are optimistically biased. Model may underperform
   on truly new data by 2-5% or more.

2. VIF REMOVED TOP PREDICTIVE FEATURES (Step 11)
   overall_sentiment_score (#2 by importance) and future_focus_ratio (#3)
   were removed because of high VIF. These features carried strong signal.
   IMPACT: Model loses access to some of its best predictors.

3. INFERENCE PIPELINE INCOMPLETE (Step 18)
   Only model + scaler + encoder are saved. All outlier transforms, encoding
   maps, and feature selection decisions are lost.
   IMPACT: Cannot deploy this model on new data.

*** HIGH ISSUES (will affect model quality) ***

4. CLASS WEIGHTS NOT APPLIED TO ALL MODELS (Steps 13/14)
   XGBoost, Gradient Boosting, and KNN don't get class weighting despite
   the pipeline deciding it's needed for this imbalanced dataset.
   IMPACT: 3 of 7 models are disadvantaged for minority classes.

5. F1_WEIGHTED METRIC FAVORS MAJORITY CLASS (Step 14)
   With 7 classes and Depression at 22.56%, f1_weighted doesn't properly
   penalize poor performance on Phobia (8.88%) or Suicidal_Tendency (8.92%).
   IMPACT: Best model might be poor at predicting the most critical classes.

6. YEO-JOHNSON OVER-APPLIED (Step 8)
   Tree-based models (which dominate this pipeline) don't need normalized
   distributions. Aggressive yeo-johnson transforms can destroy useful
   signal patterns that trees would naturally exploit.

*** MEDIUM ISSUES ***

7. EDA runs on transformed data, not raw data (Step 10)
8. No index reset after row removal — fragile for other datasets (Step 6)
9. Scaler selection uses wrong feature count (Step 12)
10. Tuning CV folds (3) differ from training CV folds (5) (Step 16)
11. drop_first=True in one-hot encoding hurts tree models (Step 9)
12. Bottom 30% feature removal is too aggressive (Step 11)

*** LOW ISSUES ***

13. warnings.filterwarnings('ignore') hides useful warnings (Step 0)
14. No early stopping for boosting models during tuning (Step 16)
15. _FixedTrial hack for model reconstruction is fragile (Step 17)
16. Task type auto-detection threshold (nunique <= 20) is arbitrary (Step 4)
17. No datetime or text feature handling (Step 9)
18. Gradient Boosting is very slow without n_jobs support (Step 14)
